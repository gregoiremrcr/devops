J1.05
Pipelines multi-stages

Principes avancés

  • Parallélisme — exécuter les jobs indépendants en parallèle
  • Cache — réutiliser les dépendances entre runs
  • Matrix builds — tester sur plusieurs versions/OS
  • Artefacts — passer des résultats entre stages
  • Environnements — staging, production avec approval gates
Règle d'or : sur une pull request, viser un feedback en moins de 10 minutes. Au-delà, optimiser.

Architecture type

jobs:
  test:        # Parallèle sur 3 versions Node
    matrix: [18, 20, 22]
    cache: npm

  build:       # Dépend de test
    needs: test
    docker: buildx + cache GHA

  security:    # Dépend de build
    needs: build
    trivy: image scan

  deploy:      # Dépend de build + security
    needs: [build, security]
    environment: production

J1.08
Terraform — Syntaxe HCL

Blocs fondamentaux

# Provider
provider "docker" {}

# Resource
resource "docker_container" "web" {
  name  = "my-app"
  image = docker_image.nginx.image_id
  ports {
    internal = 80
    external = 8080
  }
}

# Variable
variable "environment" {
  type    = string
  default = "dev"
}

# Output
output "url" {
  value = "http://localhost:8080"
}

Cycle de vie

$ terraform init
Initializing provider plugins...

$ terraform plan
Plan: 3 to add, 0 to change, 0 to destroy

$ terraform apply
Apply complete! Resources: 3 added

$ terraform destroy
Destroy complete! Resources: 3 destroyed
Idempotent : relancer apply sans changement → aucune action

Terraform — Modules & Remote State

Modules réutilisables

modules/
  webapp/
    main.tf       # Resources
    variables.tf  # Inputs
    outputs.tf    # Outputs

# Utilisation
module "webapp" {
  source      = "./modules/webapp"
  app_name    = "my-app"
  environment = "dev"
  replicas    = 3
}

Un module = un composant réutilisable avec une interface claire (variables → outputs).

Remote State

terraform {
  backend "s3" {
    bucket = "my-tf-state"
    key    = "prod/terraform.tfstate"
    region = "eu-west-1"
    encrypt = true
    dynamodb_table = "tf-lock"
  }
}
State = données sensibles. Toujours chiffrer, jamais dans Git. Utiliser un backend distant avec locking (S3 + DynamoDB, GCS, etc.).

J1.11
Ansible — Playbooks & Roles

Playbook type

---
- name: Configurer le serveur web
  hosts: webservers
  become: true

  handlers:
    - name: restart nginx
      service:
        name: nginx
        state: restarted

  tasks:
    - name: Installer Nginx
      apt:
        name: nginx
        state: present

    - name: Déployer la config
      template:
        src: nginx.conf.j2
        dest: /etc/nginx/nginx.conf
      notify: restart nginx

Structure en roles

roles/
  nginx/
    tasks/main.yml
    handlers/main.yml
    templates/nginx.conf.j2
    defaults/main.yml

# Playbook principal
- hosts: webservers
  roles:
    - base
    - nginx
Terraform + Ansible = combo gagnant.
Terraform provisionne l'infra → Ansible configure les machines.

J2.01
Docker avancé — Optimisation

Multi-stage builds

# Stage 1 : Build
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm test

# Stage 2 : Production
FROM node:20-alpine
USER node
COPY --from=builder /app/src ./src
EXPOSE 3000
HEALTHCHECK --interval=30s \
  CMD wget -q --spider http://localhost:3000
CMD ["node", "src/index.js"]

Checklist sécurité

  • Base minimale — choisir l'image la plus petite adaptée au runtime (alpine, slim, distroless...)
  • User non-root — USER node obligatoire
  • HEALTHCHECK — pour l'orchestrateur
  • .dockerignore — exclure node_modules, .git, .env
  • Pas de :latest — toujours un tag versionné
Résultat : 500 MB → 80 MB, surface d'attaque réduite de 90%

Kubernetes — Ressources essentielles

Manifeste Deployment

apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: my-app
  template:
    spec:
      containers:
        - name: app
          image: my-app:v1.2.0
          ports:
            - containerPort: 3000
          resources:
            requests:
              cpu: 100m
              memory: 128Mi
            limits:
              cpu: 250m
              memory: 256Mi
          readinessProbe:
            httpGet:
              path: /health
              port: 3000

Scaling & Rolling Updates

# Scaler à 5 replicas
$ kubectl scale deploy my-app --replicas=5

# Mettre à jour l'image
$ kubectl set image deploy/my-app app=my-app:v2

# Suivre le rollout
$ kubectl rollout status deploy/my-app

# Rollback si problème
$ kubectl rollout undo deploy/my-app
Rolling Update : maxSurge=1 et maxUnavailable=0 réduisent fortement le risque d'interruption, à condition d'avoir des probes fiables et assez de capacité.

Helm & Kustomize

Helm — Package manager K8s

# Créer un chart
helm create my-app/

# Installer
helm install my-release ./my-app \
  -f values-prod.yaml \
  -n production

# Upgrade
helm upgrade my-release ./my-app \
  --set image.tag=v2.0.0

# Rollback
helm rollback my-release 1

Chart = templates + values par défaut. Chaque env a son values-{env}.yaml.

Kustomize — Overlays natifs

base/
  deployment.yaml
  service.yaml
  kustomization.yaml
overlays/
  dev/
    kustomization.yaml   # namePrefix: dev-
    replicas-patch.yaml  # replicas: 1
  prod/
    kustomization.yaml   # namePrefix: prod-
    replicas-patch.yaml  # replicas: 5
Helm vs Kustomize : Helm pour le templating complexe et les charts communautaires. Kustomize pour les overlays simples — intégré à kubectl.

Prometheus — Métriques & Alerting

Architecture

Prometheus Server
Scrape + Stockage + Requêtes
App /metrics
Node Exporter

PromQL essentiel

# Requêtes par seconde
rate(http_requests_total[5m])

# Latence P95
histogram_quantile(0.95,
  rate(http_duration_bucket[5m]))

# Taux d'erreur
sum(rate(http_requests{status=~"5.."}[5m]))
/ sum(rate(http_requests[5m]))

Alerting

groups:
  - name: app
    rules:
      - alert: HighLatency
        expr: |
          histogram_quantile(0.95,
            rate(http_duration_bucket[5m])
          ) > 0.5
        for: 2m
        labels:
          severity: warning
        annotations:
          summary: "P95 > 500ms"
AlertManager route les alertes : Slack, PagerDuty, email. Grouping pour éviter le spam.

OPA — Politiques as Code

package kubernetes

# Interdire les conteneurs privilégiés
deny[msg] {
  container := input.spec.template
    .spec.containers[_]
  container.securityContext.privileged
  msg = sprintf(
    "%s ne doit pas être privilégié",
    [container.name])
}

# Exiger des resource limits
deny[msg] {
  container := input.spec.template
    .spec.containers[_]
  not container.resources.limits
  msg = sprintf(
    "%s doit avoir des limits",
    [container.name])
}

GitOps — Git comme source de vérité

Principes GitOps

  • Déclaratif — l'état souhaité est dans Git
  • Versionné — chaque changement est un commit
  • Pull automatique — l'agent sync le cluster
  • Self-healing — dérive détectée → correction auto
Push vs Pull : Push (CI pousse vers K8s) = credentials CI → risque. Pull (ArgoCD dans le cluster lit Git) = plus sûr.
